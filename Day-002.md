# 🎯 Goal

Understand how WordPress boots after `wp-config.php` and why every file is loaded in a specific order.

---

# 📚 Topics Covered

## 1. WordPress Bootstrap Flow

```text
Browser
    ↓
Apache / Nginx
    ↓
index.php
    ↓
wp-blog-header.php
    ↓
wp-load.php
    ↓
wp-config.php
    ↓
wp-settings.php
```

`wp-settings.php` is responsible for bootstrapping the entire WordPress application.

---

# 2. wp-settings.php

This is one of the most important files in WordPress.

Its responsibility is to initialize WordPress by loading:

- Core files
- Constants
- Cache
- Plugins
- Themes
- Lifecycle hooks

Everything depends on the order in which these files are loaded.

---

# 3. Core Files Loaded During Bootstrap

## version.php

Purpose:

- Stores WordPress version
- Database version
- TinyMCE version

WordPress loads this first because other components may depend on version information.

---

## compat.php

Purpose:

Provide compatibility functions for different PHP versions.

Example:

If a PHP function is unavailable in an older version, WordPress provides its own implementation.

---

## load.php

Purpose:

Contains helper functions required during the bootstrap process.

Examples include:

- Maintenance mode
- Memory limit
- Environment checks

These functions are needed before plugins and themes are loaded.

---

## default-constants.php

Purpose:

Defines default values for WordPress constants.

Example:

```php
WP_MEMORY_LIMIT
```

If the developer does not define a constant inside `wp-config.php`, WordPress uses the default value.

---

## plugin.php

Purpose:

Provides the Hook API.

Functions include:

```php
add_action()
add_filter()
do_action()
apply_filters()
```

Plugins depend on these functions.

Therefore this file **must be loaded before plugins**.

---

## formatting.php

Purpose:

Contains formatting and sanitization functions.

Examples:

```php
sanitize_text_field()
sanitize_email()
esc_html()
```

Plugins may use these functions during bootstrap, so they must be loaded before plugins execute.

---

# 4. Object Cache

Object Cache is initialized before plugins.

Reason:

Plugins often call functions such as:

```php
get_option()
```

If the cache already contains the data:

```text
Plugin
    ↓
Cache
    ↓
Return Data
```

No database query is required.

If Object Cache were loaded after plugins:

```text
Plugin
    ↓
Database Query
    ↓
Object Cache
```

The cache would not help during bootstrap.

---

# 5. Drop-ins

Special files loaded before normal plugins.

Examples:

```text
advanced-cache.php
object-cache.php
db.php
sunrise.php
```

Purpose:

Replace or extend WordPress core behavior very early during bootstrap.

Example:

Redis Object Cache replaces the default object cache implementation using `object-cache.php`.

---

# 6. advanced-cache.php

Purpose:

Page Cache.

If cached HTML already exists:

```text
Browser
    ↓
advanced-cache.php
    ↓
Return Cached HTML
```

WordPress does not continue bootstrapping.

Benefits:

- No database queries
- No plugins
- No themes
- Very fast response

---

# 7. MU Plugins (Must Use Plugins)

Location:

```text
wp-content/mu-plugins/
```

Characteristics:

- Automatically loaded
- No activation required
- Cannot be deactivated from the admin dashboard

Common use cases:

- Security
- Logging
- Monitoring
- Company-wide functionality

---

# 8. Network Plugins

Available only in WordPress Multisite.

Network plugins are activated once and loaded across every site in the network.

---

# 9. Regular Plugins

Loaded from:

```text
wp_options

↓

active_plugins
```

Examples:

- WooCommerce
- ACF
- Rank Math
- Contact Form 7

---

# 10. plugins_loaded Hook

Runs after every active plugin has finished loading.

Example:

```php
add_action('plugins_loaded', function () {

});
```

Typical use cases:

- Plugin integrations
- Check if another plugin exists

Example:

```php
class_exists('WooCommerce')
```

---

# 11. Theme Loading

After plugins finish loading, WordPress starts loading the active theme.

---

# 12. after_setup_theme

Purpose:

Register theme features.

Examples:

```php
add_theme_support()

register_nav_menus()

add_editor_style()
```

This hook is only for theme configuration.

---

# 13. init

Purpose:

General WordPress initialization.

Typical tasks:

- Register Custom Post Types
- Register Taxonomies
- Register Rewrite Rules
- Initialize plugin functionality

Example:

```php
register_post_type()

register_taxonomy()

add_rewrite_rule()
```

---

# 14. wp_loaded

Runs after WordPress has completely finished loading.

At this point:

- Plugins loaded
- Theme loaded
- Core initialized
- User authenticated
- Rewrite rules available

Everything is ready.

---

# 🧠 Lifecycle Order

```text
plugins_loaded
        ↓
after_setup_theme
        ↓
init
        ↓
wp_loaded
```

---

# 🎯 Responsibilities of Each Hook

| Hook | Responsibility |
|------|----------------|
| plugins_loaded | Plugin initialization and plugin integrations |
| after_setup_theme | Theme configuration |
| init | Register CPTs, Taxonomies, Rewrite Rules |
| wp_loaded | Everything has finished loading |

---

# 💡 Important Architectural Concepts

## Single Entry Point

All frontend requests go through:

```text
index.php
```

This allows WordPress to:

- Bootstrap consistently
- Load plugins
- Load themes
- Build the main query
- Render the correct template

---

## Dependency-Based Loading

WordPress does not load files randomly.

Every file is loaded because another component depends on it.

Examples:

```text
plugin.php
        ↓
Plugins
```

```text
formatting.php
        ↓
Plugins
```

```text
Object Cache
        ↓
Plugins
```

Understanding dependencies is more important than memorizing the loading order.

---

# ⭐ Interview Takeaways

- Plugins provide functionality; themes provide presentation.
- Hook API must exist before plugins load.
- Object Cache must exist before plugins to reduce database queries during bootstrap.
- Theme features belong in `after_setup_theme`.
- CPTs, Taxonomies, and Rewrite Rules belong in `init`.
- `plugins_loaded` is ideal for plugin-to-plugin communication.
- WordPress follows a dependency-driven bootstrap process.
